
# FAILOG

실패 기록을 행동 개선으로 연결하는 Streamlit 앱.

---

## 1) 프로젝트 목적 (What / Why)
- 문제 정의
  - 일반 TODO 앱은 “완료 여부”만 남고 실패 원인이 축적되지 않음
  - 실패가 반복돼도 원인 패턴(요일/유형/문구)이 구조적으로 드러나지 않음
- FAILOG 목표
  - 실패를 감정 평가가 아니라 **행동 데이터**로 취급
  - 루프: `기록 → 패턴 분석 → 코칭/실험 → 재실행`
  - 계획 문장을 더 작고 실행 가능하게 재작성하도록 유도
- 핵심 성공지표(권장)
  - 기록 지속률(DAU/WAU)
  - 실패 사유 입력률(실패 건 중 사유 포함 비율)
  - AI 코칭 실행률(코칭 버튼 클릭/노출 대비)
  - 코칭 후 계획 재작성률

---

## 2) 로컬 실행 (Getting Started)
- 요구사항
  - Python 3.10+
  - pip
- 설치/실행
  - `pip install -r requirements.txt`
  - `streamlit run app.py`
- 실행 후 확인 포인트
  - `planner.db` 자동 생성 여부
  - Planner / Failure Report / Puzzle 탭 전환 정상 동작 여부
  - 실패 원인 저장 후 Failure Report에서 데이터 반영 여부
- PDF 관련 주의
  - PDF 한글 깨짐 방지: `fonts/NanumGothic-Regular.ttf` 포함 권장

### 빠른 자체 점검 
- 1: 계획 1개 추가
- 2: 성공/실패 상태 1회 변경
- 3: 실패 사유 1문장 저장
- 4: Failure Report에서 주간 시각화 확인
- 5: Puzzle 화면 진입 및 진행도 확인

---

## 3) 사용자 플로우 (Daily / Weekly / Ops)
- 일일 플로우
  - Planner에서 계획/습관 등록
  - 수행 후 `success` 또는 `fail` 상태 업데이트
  - 실패 선택 시 원인 1문장 기록 (분석 품질에 직접 영향)
- 주간 플로우
  - Failure Report에서 요일/원인 패턴 확인
  - “주간 실험 1개”만 채택 (과부하 방지)
  - Planner 복귀 후 계획 문장 축소/구체화
- 운영 플로우
  - 카테고리 맵 갱신 주기 관리
  - AI 파싱 실패율 모니터링
  - 코칭 결과의 실행 가능성(시간/수량/조건 포함 여부) 점검

### 운영자가 보는 이상 징후
- 실패 사유 입력률 급락
- AI 호출은 있는데 코칭 적용률 낮음
- 특정 요일 실패 급증(정책/루틴 변화 신호)

---

## 4) 코드 구조 (책임 분리 / Ownership)
- 진입/라우팅
  - `app.py`
    - 앱 엔트리
    - 페이지 기본 설정
    - 화면 라우팅 및 공통 패널 구성
  - `failog/nav.py`
    - 상단 네비게이션 상태 관리
- 화면 모듈
  - `failog/screens_planner.py`
    - 계획/습관 CRUD UI
    - 리마인더 토스트
    - 위험도/AI rewrite 연동
  - `failog/screens_failures.py`
    - 대시보드 시각화
    - 원인 분석/코칭/챗/실험
    - PDF 리포트
  - `failog/screens_puzzle.py`
    - 퍼즐 진행도 렌더
    - 보관함 표시
- 도메인 모듈
  - `failog/habits_tasks.py`: tasks/habits CRUD + 상태 로직
  - `failog/puzzle.py`: 조각 지급/완성 저장/보관함 적재
  - `failog/risk.py`: 위험도 계산(규칙 + AI)
  - `failog/coaching.py`: 코칭/실험/챗 프롬프트 및 호출
  - `failog/categorization.py`: 실패 원인 카테고리화
- 인프라
  - `failog/db.py`: SQLite 연결/초기 스키마
  - `failog/prefs.py`, `failog/cookies.py`: 쿠키 기반 설정 저장

### 변경 시 기본 원칙
- 화면 파일에서 DB 쿼리를 직접 늘리지 말고 도메인 계층으로 위임
- AI 호출 로직은 `coaching.py`/`risk.py`에서 일관 관리
- DB 스키마 변경 시 기존 데이터 호환 우선

---

## 5) 데이터 모델 / DB 운영
- 핵심 테이블
  - `habits`: 반복 습관 정의
  - `tasks`: 날짜별 항목/상태/실패원인 (`todo/success/fail`)
  - `category_maps`: 실패 원인 분류 맵(LLM 결과 저장)
  - `puzzle_state`: 현재 퍼즐 상태(공개 조각/완료일 등)
  - `puzzle_gallery`: 완성 퍼즐 보관 내역
- 운영 규칙
  - 스키마 변경: 하위호환 우선(누락 컬럼 보강)
  - 백업: `planner.db` 일 단위 백업 권장
  - 복구 리허설: 월 1회 이상 권장
- 정합성 점검(권장)
  - 비정상 상태값 존재 여부
  - 중복 레코드/고아 데이터
  - 최근 데이터 급감(수집 누락 가능성)

### 권장 운영 체크리스트
- 배포 전
  - 스키마 변경사항 문서화
  - 로컬 샘플 DB 마이그레이션 테스트
- 배포 후
  - 최근 24시간 에러/파싱 실패율 확인
  - 핵심 사용자 플로우(계획→실패→리포트) 스모크 테스트

---

## 6) AI 실행 정책 / Token Governance
- 활성 조건
  - 사용자 동의(consent)
  - API Key 설정 완료
- 호출 원칙
  - 버튼 기반 명시 호출 우선
  - 자동 대량 호출 지양
  - 필요한 최소 컨텍스트만 전송
- 출력 안정성
  - JSON 출력 강제
  - 파싱 실패 시 폴백 응답
  - 모델 오류 시 사용자 안내 + 기본 기능 유지
- 비용 통제
  - 입력 길이 상한(최근 N개)
  - 기능별 모델/온도 분리
  - 월간 사용량 임계치 모니터링 권장

### 프롬프트 품질 기준(운영)
- 비난/자책 유도 금지
- 추상적 문장보다 행동 규칙(시간/수량/조건) 우선
- 반복 실패 원인에는 다른 각도의 대안 포함

## 7) 로컬 보안 정책
- 비밀정보
  - OpenAI API Key 저장소 커밋 금지
  - `.env`/시크릿/쿠키 사용 시 팀 정책에 맞춰 분리 관리
  - 공용 기기에서는 쿠키 저장 비활성 권장
- 민감데이터
  - 실패 원문은 민감 맥락 가능 → 로그/공유 최소화
  - `uid` 링크 공유 시 접근 리스크 반드시 고지
- 릴리즈 규율
  - 프롬프트 변경은 단계 반영
  - 변경 전/후 품질지표 비교 기록

### 사고 대응
- 키 유출 의심: 즉시 키 폐기/재발급
- 민감 데이터 노출: 공유 링크 비활성/데이터 접근 차단/원인 분석

---

## 8) 기기 연속 사용·데이터 이전 안내
- 저장 위치 이해
  - 핵심 데이터: `planner.db`
  - 일부 설정: 브라우저 쿠키(기기 간 자동 승계 안 될 수 있음)
- 이전 체크리스트
  1. `planner.db` 백업 확보
  2. 대상 코드 버전/스키마 호환 확인
  3. API Key/리마인더 등 쿠키 설정 재입력
  4. `uid` 링크 공유 정책 재확인
- 검증 체크
  - 이전 후 Planner 과거 데이터 조회 가능 여부
  - Failure Report 집계 일관성 확인
  - Puzzle 진행 상태/보관함 노출 확인
- 중장기 개선
  - 계정 기반 인증 도입 검토
  - 사용자 단위 접근 제어/감사 로그 체계화
