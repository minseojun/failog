# FAILOG
FAILOG는 **실패 기록을 실행 가능한 개선 행동으로 전환**하는 Streamlit 애플리케이션입니다.  
핵심은 `기록 → 패턴 분석 → 코칭/실험 → 재실행` 루프를 짧게 반복하는 것입니다.

---

## 빠른 시작

### 1) 설치 및 실행

```bash
pip install -r requirements.txt
streamlit run app.py
```

### 2) 첫 실행 체크포인트

- 앱 실행 시 로컬 SQLite 파일 `planner.db`가 자동 생성됩니다.
- PDF 한글 렌더링 안정성을 높이려면 `fonts/NanumGothic-Regular.ttf`를 저장소에 포함하세요.

### 3) 초기 사용자 루틴

1. Planner에서 오늘 계획/습관을 등록
2. 성공/실패 체크 및 실패 사유 간단 기록
3. Failure Report에서 주간 패턴/코칭 확인
4. 퍼즐 보상으로 기록 루틴 유지

## 2) 사용자 사용 시나리오 (User Playbook)

- 완료/실패 상태 업데이트
- 실패 원인 기입
- Failure Report에서 반복 패턴 확인

### B. 집중 개선 사용자

- 위험도 확인 후 AI rewrite 적용
- “이번 주 실험 1개”만 채택해 과부하 방지
- 2주 단위로 동일 실패 재발 여부 점검

### C. 운영/코치 역할

- 카테고리 맵 갱신 주기 설정
- 코칭 생성 품질(구체성/재현성) 점검
- 프롬프트 출력 형식(JSON) 실패율 모니터링

## 3) 기능·도메인 책임도 (Module Responsibility Map)

- `app.py`  
  앱 엔트리, 화면 라우팅, 글로벌 UI 진입점

- `failog/screens_planner.py`  
  계획/습관 입력, 알림 토스트, 위험도/AI rewrite 연계

- `failog/screens_failures.py`  
  대시보드, 주간 분석, 코칭 챗, 실험 제안, PDF 리포트

- `failog/screens_puzzle.py` + `failog/puzzle.py`  
  퍼즐 상태/조각 지급/완성 보관함

- `failog/coaching.py`, `failog/risk.py`, `failog/categorization.py`  
  AI 프롬프트/분석/코칭/카테고리화 계층

- `failog/db.py`, `failog/habits_tasks.py`  
  데이터 저장/조회 및 스키마 초기화

### 데이터 흐름 (요약)

`사용자 입력 → tasks/habits 저장 → 실패 누적 → AI 분석 요청(옵트인) → 행동 대안 제시 → 재입력`

## 4) 데이터 계층 & DB 운영 규약 (Data & DB Ops)

### 핵심 테이블

- `habits`: 반복 습관 정의
- `tasks`: 날짜별 할 일/상태/실패사유
- `category_maps`: 실패 원인 카테고리 맵
- `puzzle_state`: 현재 퍼즐 진행 상태
- `puzzle_gallery`: 완성 퍼즐 보관함

### 운영 규칙

- 스키마 변경 시: 기존 DB와의 하위호환(누락 컬럼 보강) 우선
- 정합성 점검: 주 1회 중복/고아 데이터 점검 쿼리 수행 권장
- 백업: 배포 환경에서는 `planner.db` 주기 백업(최소 일 단위)

### 보관 정책 권장안

- 실패 원문 텍스트는 민감정보 가능성이 있으므로 접근권한 최소화
- 필요 시 보관 기간(예: 6~12개월) 정책을 별도 문서로 명시

---

## 5) AI 실행 체계와 프롬프트 거버넌스 (AI Runtime Governance)

### 호출 원칙

- 기본적으로 사용자 명시 행동(버튼/요청) 시 호출
- 동의(consent)와 API Key가 없으면 AI 기능 비활성
- 출력 형식은 JSON 고정으로 파싱 안정성 확보

### 토큰/비용 관리 정책

- 입력 길이 상한 설정(최근 데이터 N개)
- 기능별 모델 분리(저비용 분석 vs 고품질 코칭)
- 월간 예산 경보 임계치 운영 권장

### 실패 대응

- JSON 파싱 실패 시 안전한 기본 응답 사용
- 모델 오류 시 사용자 안내 메시지 제공
- 로그에는 프롬프트 원문 전체 저장 지양(민감정보 최소화)

---

## 6) 로컬 정책/보안/비밀정보 처리 (Local Security Policy)

### 비밀정보

- OpenAI API Key는 코드/저장소 커밋 금지
- 브라우저 쿠키 저장은 선택 기능이며, 공유 기기에서는 비활성 권장

### 개인정보/민감정보

- 실패 사유 텍스트는 개인 맥락이 포함될 수 있음
- 운영 시 접근권한, 백업 암호화, 최소 로그 원칙 적용

### 릴리즈 정책

- 프롬프트/코칭 로직 변경은 실험 플래그 또는 단계적 배포 권장
- 변경 시 “전/후 품질 비교 지표”를 기록

---

## 7) 멀티디바이스 이전 전략 (Cross-Device Migration)

### 현재 구조 이해

- 사용자 구분은 주로 `uid` 및 저장 데이터 문맥에 의존
- 쿠키 기반 설정(OpenAI 키, 일부 환경설정)은 기기별로 독립

### 이전 체크리스트

- 데이터 이전: `planner.db` 백업/복원 절차 수립
- 설정 이전: 키/모델/알림 설정 재입력 필요 여부 안내
- 링크 공유: `uid` 노출 시 데이터 접근 리스크 고지

### 중장기 권장

- 계정 기반 인증 도입(선택)
- 암호화된 서버 저장소 + 사용자별 권한 분리

## 8) 프론트엔드 변경 실행안 (UI Delivery Blueprint)

변경 요청이 들어오면 아래 순서를 따릅니다.

1. 사용자 문제를 1문장으로 정의
2. 영향 화면(Planner/Failure Report/Puzzle) 식별
3. 상태 영향(`session_state`, 쿠키, DB) 확인
4. UI 가독성/접근성/모바일 폭 검증
5. 회귀 테스트 시나리오 작성
6. 롤백 기준/방법 기록

### 완료 조건 (DoD)

- 기능 동작 + 시각적 회귀 없음
- 최소 1개 사용자 시나리오 수동 검증
- 변경 이력 README 또는 별도 CHANGELOG 반영

## 9) 일회성 점검 프레임워크 (One-time Diagnostic Kit)

### 언제 수행하나

- 릴리즈 직전
- 주요 프롬프트 개편 직후
- 지표 급변(이탈/실패율 증가) 시점

### 점검 항목

- 성능: 첫 화면 로드/AI 응답 체감 시간
- 품질: AI 응답 파싱 성공률, 빈 응답률
- 데이터: 최근 4주 입력량/실패원인 입력률
- UX: 핵심 버튼 클릭 퍼널(작성→체크→분석)

### 산출물 템플릿

- 현황 요약 (1페이지)
- 즉시 수정 3건 (Hotfix)
- 중기 개선 3건 (Roadmap)

## 10) 확산/정착 전략 (Adoption & Skill Promotion)

### 대상별 메시지

- 신규 사용자: “실패 기록만 해도 다음 행동이 쉬워진다”
- 파워 사용자: “패턴 분석으로 반복 실패를 줄인다”
- 운영자: “코칭 품질을 데이터 기반으로 관리한다”

### 운영 지표

- 주간 활성 사용자
- 실패 원인 입력률
- AI 코칭 실행률
- 코칭 이후 계획 재작성률

### 실행 방식

- 소규모 사용자 그룹 베타 운영
- 온보딩 문구/튜토리얼 A/B 테스트
- 성공 사례(전/후 변화) 축적

## 11) 장애 대응 및 FAQ

### 자주 발생하는 이슈

- **AI가 동작하지 않아요**  
  → 동의 체크, API Key 입력, 모델명 확인

- **설정이 저장되지 않아요**  
  → 브라우저 쿠키 차단 여부 확인

- **PDF 한글이 깨져요**  
  → 한글 폰트 파일 포함 여부 확인

- **알림이 안 와요**  
  → 이 앱은 푸시 알림이 아닌 인앱 토스트 중심 동작(앱 열림 상태 필요)
